## TL;DR

- Le contenu est du **markdown versionné** dans Git — chaque article est un fichier.
- Hermes écrit le fichier, le commit, le pousse ; **Vercel reconstruit** automatiquement.
- Aucune base de données, aucun CMS. Le dépôt *est* la source de vérité.

## L'architecture

Le principe est volontairement simple :

```text
Agent Hermes (Mac, chaque matin)
   └── écrit content/blog/AAAA-MM-JJ-slug.md
        └── git commit + git push
             └── Vercel détecte le push
                  └── rebuild Next.js → site en ligne
```

Pas d'intermédiaire. Le contenu vit à côté du code, dans le même dépôt, avec le même historique.

## La stack

- **Next.js** (App Router) pour le rendu et le routage.
- **gray-matter** pour lire les métadonnées en tête de chaque fichier.
- Un pipeline **remark / rehype** pour transformer le markdown en HTML, avec coloration syntaxique.
- **Vercel** pour l'hébergement et le déploiement continu.

## Pourquoi du markdown plutôt qu'un CMS

Trois raisons. D'abord la **portabilité** : un fichier texte se lit partout, se sauvegarde partout, survit à n'importe quelle plateforme. Ensuite la **traçabilité** : chaque modification est un commit, daté et signé. Enfin la **simplicité d'écriture pour un agent** : produire un fichier markdown est trivial et sans effet de bord.

> La meilleure infrastructure est celle qu'on oublie. Ici, publier, c'est écrire un fichier.
