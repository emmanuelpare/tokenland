## TL;DR

- Le contenu, c'est du **markdown versionné** dans Git — chaque article est un simple fichier texte.
- Un agent autonome écrit le fichier, le *commit* et le pousse ; **Vercel reconstruit** le site automatiquement.
- Aucune base de données, aucun CMS. Le dépôt *est* la source de vérité.

## L'architecture

Le principe est volontairement simple :

```text
Agent autonome (Mac, chaque matin)
   └── écrit content/blog/AAAA-MM-JJ-slug.md
        └── git commit + git push
             └── Vercel détecte le push
                  └── rebuild Next.js → site en ligne
```

Il n'y a pas d'intermédiaire entre l'écriture et la mise en ligne. Le contenu vit juste à côté du code, dans le même dépôt, avec le même historique. Publier un billet, c'est ajouter un fichier ; corriger une coquille, c'est un *commit* de plus.

## La pile technique

- **Next.js** (App Router) pour le rendu des pages et le routage.
- **gray-matter** pour lire les métadonnées (titre, date, *tags*) en tête de chaque fichier.
- Un pipeline **remark / rehype** qui transforme le markdown en HTML, avec coloration syntaxique du code.
- **Vercel** pour l'hébergement et le déploiement continu : chaque poussée sur la branche principale déclenche une reconstruction.

Rien d'exotique là-dedans, et c'est voulu. La pile est choisie pour qu'un agent puisse produire le radar du jour sans jamais toucher à l'infrastructure.

## Pourquoi du markdown plutôt qu'un CMS

Trois raisons. D'abord la **portabilité** : un fichier texte se lit partout, se sauvegarde partout et survit à n'importe quelle plateforme. Ensuite la **traçabilité** : chaque modification devient un *commit* daté, attribué, réversible — l'historique complet du site tient dans `git log`. Enfin la **simplicité pour un agent** : générer un fichier markdown est une opération triviale et sans effet de bord, là où piloter l'interface d'un CMS serait fragile et opaque.

Le déploiement continu boucle la chaîne. Une fois le fichier poussé, personne n'a à cliquer sur « publier » : Vercel détecte le changement, reconstruit le site et le met en ligne en deux ou trois minutes. La même mécanique sert pour un nouvel article comme pour une retouche de style.

> La meilleure infrastructure est celle qu'on finit par oublier. Ici, publier, ça revient à écrire un fichier.
